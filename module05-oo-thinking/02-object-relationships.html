Object Relationships ‚Äî The Big Picture
üè´ The Professor Introduces Relationships
üë®‚Äçüè´
Professor
Today we will learn something very important, which is called relationships. I will make it simple for you. There are three relationships we will learn. There is a Dependency relationship, there is Aggregation, and there is Inheritance. Very simple!

Dependency says "uses." Aggregation says "has." Inheritance says "is."
üîß
Dependency
"uses"
Class A temporarily uses Class B (as a method parameter or local variable). B is not part of A's identity ‚Äî just a tool.
üì¶
Aggregation
"has"
Class A has a Class B stored as a field. Comes in two strengths: Aggregation (weak) and Composition (strong).
üß¨
Inheritance
"is"
Class A is a type of Class B. Studied in Module 06.
üìå Aggregation has two strengths

Both Aggregation and Composition are "has-a" relationships ‚Äî they both mean one class stores another as a field. The difference is the lifetime of the contained object.

üì¶
Aggregation
"has-a" (weak)
The contained object can exist independently. It can be shared among multiple owners. Open diamond ‚óá in UML.
üíé
Composition
"owns-a" (strong)
The contained object cannot exist without its owner. It dies when the owner dies. Filled diamond ‚óÜ in UML.

üë®‚Äçüè´

2
Dependency ‚Äî "uses"

To decide what kind of relationship BMICalculator has with Student, we do not start with the code ‚Äî we start with the model. We ask one philosophical question: "Is the BMICalculator part of the student's identity?" The answer to that question determines everything.

üè´ Step 1 ‚Äî The Identity Question: What defines a student?
üë®‚Äçüè´
Professor
Let me ask you this: what defines a student? What are the things that make this student who he is?
üôã
Students
His name, his ID number, his address, his marks!
üë®‚Äçüè´
Professor
Exactly. Name, ID, address, marks ‚Äî these are the things that constitute the identity of the student. They describe who he is. Now ‚Äî does the BMICalculator define him? Is it part of who he is?
üôã
Students
No‚Ä¶ the BMICalculator is not who he is. It is just something he uses.
üë®‚Äçüè´
Professor
Correct! Now imagine the BMICalculator is a device ‚Äî a machine standing in the corridor of the university. The student walks up to it, enters his weight and his height, gets his BMI value, and walks away. The machine is not part of him. He does not carry it. He does not own it. Any student can use it. It has nothing to do with his identity.
üß† Step 2 ‚Äî Philosophy First, Code Second
üë®‚Äçüè´
Professor
So the decision is made at the model level, not the code level. Because BMICalculator is not part of the student's identity, we do not store it as a data field inside the student. We use it only when we need it ‚Äî we pass it as a method parameter. That is what you will see in the code. But the reason is philosophical ‚Äî not syntactic.
üôã
Student
Professor ‚Äî but what if I did write it as a data field? Would it then become aggregation?
üë®‚Äçüè´
Professor
Great question ‚Äî and the answer depends on the real world. Ask yourself: does every student own a personal BMICalculator device? If every student carried their own personal machine ‚Äî one that belongs to them and no one else ‚Äî then yes, it would make sense to store it as a field, and the relationship would be aggregation. But look at what BMICalculator actually is. It holds no student data. It has no connection to any particular student. It is a shared tool ‚Äî like the machine in the corridor that any student can walk up to and use. No student owns one. No student carries one as part of who they are. So if you write it as a field, the code will compile ‚Äî but you have described something that is not true in the real world. The code is supposed to reflect the model. If the code contradicts the model, the code is wrong.
üôã
Students
So the rule is: if it is not part of identity ‚Üí do not make it a field ‚Üí use it as a parameter ‚Üí and that is dependency!
üë®‚Äçüè´
Professor
Exactly! ‚úÖ And notice ‚Äî we can even go further and make getBMI() a static method. Because the BMICalculator has no connection to any specific student, we do not even need an object ‚Äî we just call BMICalculator.getBMI(weight, height) directly. This is the most natural way to express that it is purely a utility tool, with no ownership and no identity relationship.

Aggregation & Composition ‚Äî "has"

Both aggregation and composition are "has-a" relationships ‚Äî the class stores another object as a data field. The key question is always: "Can the contained object survive without the owner?"

üè´ Step 1 ‚Äî What does "Aggregation" mean?
üë®‚Äçüè´
Professor
One question ‚Äî what does "aggregation" mean? We have two terms to understand: the aggregating class and the aggregated class.

Aggregation means aggregating ‚Äî collecting things together. In our case, the Student class is the aggregating class ‚Äî it collects other objects inside it. The Address and Name are the aggregated classes ‚Äî they are collected inside the student.
üè† Step 2 ‚Äî The Apartment Analogy: Composition vs Aggregation
üë®‚Äçüè´
Professor
Let me ask you a very simple question. Can you imagine a Name without a Student? Can we say "this is Ali Fahad" ‚Äî but where is he? He belongs to what? Who is this name for? There is no student ‚Äî there is no name!
üôã
Student
That is true, Professor. "Ali Fahad" cannot exist by itself without belonging to someone. The name only makes sense if there is a person ‚Äî a student!
üë®‚Äçüè´
Professor
Now imagine three students ‚Äî Ali, Khalid, and Fahad ‚Äî living in the same apartment on Al-Jami'a Street, Unit 100, Jeddah. They all share the same address.

Now Ali leaves the university. What happens to the address "Al-Jami'a Street, Unit 100, Jeddah"? Does it disappear?
üôã
Students
No! The address still exists ‚Äî Khalid and Fahad still live there! The address survived even though Ali left.
üë®‚Äçüè´
Professor
Exactly! ‚úÖ But if Ali died ‚Äî then the name "Ali Fahad" must die with him. Can you see the relationship?

üîµ Address ‚Üí Aggregation (weak): the address can be shared and still exists independently of any single student.
üíö Name ‚Üí Composition (strong): the name cannot exist without its student ‚Äî it dies with the student.
üôã
Students
Wow! üòÆ And the BMICalculator ‚Äî it is not part of the student at all, right? It is not aggregation either. It is just used temporarily ‚Äî dependency!
üë®‚Äçüè´
Professor
Perfect! The BMI calculator is only a tool used temporarily to calculate a value ‚Äî it does not contribute to the identity or definition of a student. That is precisely why it is dependency, not aggregation. üéâ

üè´ Step 3 ‚Äî The Student's Challenge: How Do I Write This in Java?
üë®‚Äçüè´
Professor
My students ‚Äî we have now studied aggregation and composition closely. You know the example: Student, Name, and Address. You know which one is composition and which one is aggregation. Good!
üôã
Student
Professor ‚Äî I understand the concept. But I have a challenge. When I go to write the Java code‚Ä¶ how do I actually make the Name object die when the Student is destroyed? And how do I make the Address stay alive? The UML shows a diamond ‚óÜ or ‚óá ‚Äî but how do I express that in code?
üë®‚Äçüè´
Professor
Excellent question ‚Äî this is exactly what you must understand to become a good programmer. The answer is one rule: it is all about where you write new.

For composition ‚óÜ ‚Äî you write new Name(...) inside the Student constructor. That Name object is created by the student, owned by the student, and has no other reference outside. When the student is gone ‚Äî the name is gone with it. No one else holds a reference to it.

For aggregation ‚óá ‚Äî you create the Address object outside, in main or somewhere else, and then you pass it in to the constructor. Now multiple students can hold a reference to the same address object. When one student is destroyed, the address is still alive ‚Äî because the other students still have their reference to it.
üôã
Students
Ahh! So the secret is where the new is written!
‚Äî new is inside the constructor ‚Üí composition ‚óÜ ‚Üí dies with the owner
‚Äî new is outside, object passed in ‚Üí aggregation ‚óá ‚Üí can survive independently
üë®‚Äçüè´
Professor
Exactly! ‚úÖ That is the single rule you must practice. Look at the code carefully ‚Äî the Student constructor receives first and last as plain String parameters, then calls new Name(first, last) right there inside. That is composition. But for address, it receives an Address object that was already created outside ‚Äî that is aggregation. One constructor, two different relationships, expressed by one small difference in where new is written.

üìÑ Student.java ‚Äî composition vs aggregation in code

‚úÖ The Key Programmer's Rule
  • Composition ‚óÜ ‚Üí create the object inside the constructor using new. The object is exclusive to this owner.
  • Aggregation ‚óá ‚Üí receive the object from outside as a constructor parameter. It can be shared.
  • Dependency - - ‚Üí ‚Üí receive as a method parameter only. Never stored as a field. Not part of the object's identity.

Self-Aggregation ‚Äî A Class that Refers to Itself

Now we move to something interesting ‚Äî self-aggregation. What does it mean? It is aggregation between objects of the same class. The class has a data field whose type is the same class. This might sound strange ‚Äî but it is very natural once you see the example!

üè´ Classroom Discussion ‚Äî "How do we assign a supervisor?"
üë®‚Äçüè´
Professor
Let me make it simple. I have three students: Ali, Khaled, and Fahad. Just imagine they have an ID and a name ‚Äî that is it. Now I want to put Ali as the supervisor of Khaled and Fahad. How should I store that information in the Student class?
üôã
Student

üë®‚Äçüè´
Professor
That could work ‚Äî but think about it. What if sometime you want to know the supervisor's age? Their GPA? Their address? Any detail about them? If you only store the ID, you would have to search through all students every time to find that information. It becomes very complicated and slow!

Multiplicity ‚Äî How Many Objects Are Involved?

Before we move to the next topic, the professor explained one more concept in UML ‚Äî multiplicity. This answers the question: "How many objects of a class are involved in a relationship?"

üè´ Classroom Discussion ‚Äî What is multiplicity?
üë®‚Äçüè´
Professor
In UML, when we draw a relationship between two classes, we can have many-to-many or one-to-many relationships. We need to show the student what this means. For each class involved in a relationship, we write a number ‚Äî or a range ‚Äî at its end of the relationship line. This is called multiplicity.
üë®‚Äçüè´
Professor
For example ‚Äî a Teacher can teach zero to three courses (0..3). Each Course has exactly one teacher (1). A course must have between 5 and 60 students (5..60). And a student can take many courses (*). This is multiplicity!
üìê What is Multiplicity?

Multiplicity specifies how many instances of one class can be associated with one instance of another class. It is written at both ends of a relationship line in UML. Each class involved in a relationship has its multiplicity specified.

NotationMeaningExample
1Exactly oneA course has exactly 1 teacher
0..1Zero or one (optional)A student may have 0 or 1 supervisor
* or 0..*Zero or moreA student can take many courses
1..*One or moreA faculty must have at least 1 teacher
5..60Between 5 and 60A course section has 5 to 60 students
0..3Zero to threeA teacher teaches 0 to 3 courses

üéì Example: Teacher ‚Üí Course ‚Üí Student

üìù How to Read Multiplicity Labels

Read each label from the perspective of the other class.

  • The 1 near Teacher means: "from a Course's point of view, it has exactly 1 teacher."
  • The 0..3 near Course means: "from a Teacher's point of view, they teach 0 to 3 courses."
  • The 5..60 near Course means: "from a Student's point of view, each course they take has 5 to 60 enrolled students."
  • The * near Student means: "from a Course's point of view, it can have any number of students."
