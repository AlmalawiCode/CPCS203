Static Method Hiding

Inherited But Not Overridden — A Subtle Distinction

You already know that instance methods can be overridden — the child replaces the parent's behaviour at runtime. But static methods follow different rules. A child can define a static method with the same name as the parent's, but what happens is called hiding, not overriding. The difference is invisible in the code — but the behaviour is completely opposite. This page is dedicated to understanding exactly why.

1 Static Methods & Inheritance — Hiding, Not Overriding

Everything you learned about overriding applies to instance methods. But what happens when a static method in the parent has the same signature as a static method in the child? The answer surprises most students: it is not overriding — it is called method hiding, and the behaviour is completely different.

▶ Instance Method — Overriding
✓ Uses dynamic binding
✓ Decision made at runtime
✓ Based on the actual object type
✓ Polymorphism — the core OOP power
▶ Static Method — Hiding
✗ Uses static binding
✗ Decision made at compile time
✗ Based on the reference type (declared type)
✗ No polymorphism — the child method just "hides" the parent's

Step 1 The Classes

class Animal { public static void classify() { // static method System.out.println("I am an Animal"); } public void speak() { // instance method System.out.println("..."); } } class Dog extends Animal { public static void classify() { // HIDES Animal.classify() System.out.println("I am a Dog"); } @Override public void speak() { // OVERRIDES Animal.speak() System.out.println("Woof!"); } }

Step 2A — Static Call Static Methods via the Class Name

Static methods belong to the class, not to any object. The correct way to call them is through the class name. Each class keeps its own separate version — they do not replace each other.

// Static methods: call via CLASS NAME Animal.classify(); // Animal's own version Dog.classify(); // Dog's own version
Output
I am an Animal I am a Dog
💡 Both versions exist independently. Dog.classify() does not replace Animal.classify() — it simply hides it. Call each one through its class name and you always get exactly what you asked for.

Step 2B — Instance Call Instance Methods via an Object — Overriding Works Differently

Instance methods are resolved using the actual object type at runtime. When Dog overrides speak(), any Dog object — no matter how it is declared — will call Dog's version. That is the power of polymorphism.

// Instance methods: call via OBJECT — actual type decides Dog d = new Dog(); d.speak(); // Dog's speak() — overrides Animal's Animal a = new Dog(); // declared as Animal, actual object is Dog a.speak(); // still Dog's speak() — real polymorphism!
Output
Woof! Woof!
✓ This Is Real Overriding

Whether the variable is declared as Dog or Animal, the actual object at runtime is Dog — so Dog.speak() always wins. The JVM looks at the real object, not the declared type.

Warning What If You Call a Static Method via an Object Reference?

Java allows you to write a.classify() even when classify() is static — but this is bad style and produces a surprising result:

Animal a = new Dog(); // actual object is Dog a.classify(); // looks like it should call Dog's classify()...
Output — Surprise!
I am an Animal
⚠ Why? Static Binding Uses the Declared Type, Not the Actual Object

The variable a is declared as Animal. For static methods, Java resolves the call at compile time using the declared type — it never looks at the real object in memory. So a.classify() is silently treated as Animal.classify().

This is exactly why calling static methods via object references is misleading and should be avoided. Always call static methods through the class name: Animal.classify() or Dog.classify().

🔑 The Rule to Remember

Instance overriding — the JVM looks at the actual object type at runtime. The child's method replaces the parent's.

Static hiding — the compiler uses the declared reference type at compile time. Both methods exist independently; neither replaces the other. Always call them through the class name to make your intention clear.

Summary Override vs Hide — Side by Side

Aspect Overriding (instance) Hiding (static)
Binding type Dynamic (runtime) Static (compile time)

Polymorphism? Yes ✓ No ✗

Challenge Overriding or Hiding?

✌ Challenge — Spot the Trap
The actual object created is a Circle. So why does s.type() not print "Circle"? What does it print, and why?
class Shape { public static String type() { return "Shape"; } public String draw() { return "Drawing Shape"; } } class Circle extends Shape { public static String type() { return "Circle"; } @Override public String draw() { return "Drawing Circle"; } } public class Main { public static void main(String[] args) { Shape s = new Circle(); // declared Shape, actual Circle System.out.println(s.type()); // static — which version? System.out.println(s.draw()); // instance — which version? } }
Output:   Shape  /  Drawing Circle
Line 1 — s.type() prints "Shape": type() is static. Even though the actual object is a Circle, the declared type of s is Shape. Java resolves the static call at compile time using the declared type → Shape.type() runs. The actual object is ignored.

Line 2 — s.draw() prints "Drawing Circle": draw() is an instance method that Circle truly overrides. Java resolves it at runtime using the actual object type → Circle.draw() runs. Polymorphism works here.

Lesson: to get Circle's type, you must write Circle.type() — not go through an object reference.
