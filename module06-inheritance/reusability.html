Reusability

The Second Power of Inheritance

In Part 1, you understood what inheritance is and why it matters. You saw how it saves code from redundancy and how a child class builds on its parent — just as Omar said: "each generation built on the one before."

Now we go deeper. A child class does not just receive — it can change what the parent gave it, and add what the parent never had. That is Reusability: the second reason we use inheritance, and the one with the most power.

In Part 1, we introduced reusability as one of the two reasons to use inheritance — and we briefly showed examples. Now this entire module is dedicated to it. Let us be very clear about what it means.

💡 Reusability — Exactly Two Ways

When a child class reuses its parent, it can do it in exactly two ways. There is no third option. Every act of reusability through inheritance is either one of these — or both at once:

🔄
Way 1
Override

Take a method the parent already has — and replace its behaviour with your own version. The method name stays the same. Only the implementation changes.

📌 You saw this with Ali's ticket report: Ahmed overrode printf() once and fixed all 11 lines instantly.
✨
Way 2
Add New Features

Give the child class something the parent never had — a new method or a new field. You are extending the parent, not replacing anything it already does.

📌 You saw this with EnhancedPrintWriter: adding print2DArray() — a method PrintWriter never had.
🌟 What This Module Covers

Now that you know the two paths, this module will take you through each one in depth — the rules, the syntax, the common mistakes, and the powerful tools Java gives you to make both work correctly. By the end, reusability will not be a word you read — it will be a skill you have.

2 Recall from CPCS201 — Method Overloading

Before introducing overriding, we must be very clear about something you already know from CPCS201: method overloading. The two words sound similar, but they describe completely different things.

📚 Overloading — The CPCS201 Rule

A class may have multiple methods with the same name, as long as they have different signatures. The signature is determined by the number, types, and order of parameters. The return type is NOT part of the signature.

class Printer { // These three are overloads of the same method name "print" public void print(String msg) { System.out.println(msg); } public void print(String msg, int times) { // different: extra parameter for (int i = 0; i < times; i++) System.out.println(msg); } public void print(int number) { // different: parameter type System.out.println("Number: " + number); } }
💡 Key Point about Overloading

Overloading happens within one class. It has nothing to do with inheritance. Java resolves which overload to call at compile time, based purely on the argument types you pass.

3 Method Overriding — Redefining Inherited Behaviour

Now the new concept. When a child class provides its own implementation of a method that already exists in the parent — with the exact same name and the exact same signature — this is called method overriding.

🌟 The Definition of Method Overriding

A method in a subclass overrides a method in its superclass when it has the identical name and the identical parameter list (same number, types, and order). When the method is called on a child object, the child's version runs — not the parent's.

Purple = original method in parent  |  Red = overridden (same signature, new body)  |  Green = brand-new method added in child

Rules The Three Rules of Overriding

1
Same Method Name

The overriding method must have the exact same name as the method in the parent class.

2
Same Signature (parameters)

The parameter list must be identical. If you change the parameters, it becomes overloading, not overriding.

3
Requires Inheritance

Overriding only makes sense when there is a parent-child relationship via extends. No inheritance = no overriding.

Complete Example Animal → Dog and Cat

Read the parent class first, then each child carefully. Notice the @Override annotation — always write it. It tells Java: "I intend to override. Tell me if I make a mistake."

// ── Parent Class ───────────────────────────── class Animal { private String name; public Animal(String name) { this.name = name; } public String getName() { return name; } public String makeSound() { return "..."; // generic — every animal makes SOME sound } public void describe() { System.out.println(name + " says: " + makeSound()); } } // ── Child Class 1 ──────────────────────────── class Dog extends Animal { private String breed; public Dog(String name, String breed) { super(name); this.breed = breed; } @Override public String makeSound() { // ← OVERRIDING Animal.makeSound() return "Woof!"; } public void fetch() { // ← NEW METHOD — only exists in Dog System.out.println(getName() + " fetches the ball!"); } } // ── Child Class 2 ──────────────────────────── class Cat extends Animal { public Cat(String name) { super(name); } @Override public String makeSound() { // ← OVERRIDING Animal.makeSound() return "Meow!"; } } // ── Demo ───────────────────────────────────── public class Main { public static void main(String[] args) { Dog d = new Dog("Rex", "Labrador"); Cat c = new Cat("Whiskers"); d.describe(); // describe() is INHERITED — but it calls Dog's makeSound() c.describe(); // same — calls Cat's makeSound() d.fetch(); // only Dog has this } }
Output
Rex says: Woof! Whiskers says: Meow! Rex fetches the ball!
💡 Notice something powerful: describe() is defined only once — in Animal. But when called on a Dog, it uses Dog's makeSound(). When called on a Cat, it uses Cat's makeSound(). This is the essence of polymorphism — which we will explore fully in Module 7.
✎ Why @Override?

The @Override annotation is optional but strongly recommended. Without it, if you accidentally misspell the method name or use the wrong signature, Java will silently treat it as a brand-new method instead of telling you the mistake. With @Override, the compiler will immediately report an error if the override doesn't match. Always use it.

4 Way 2 — Adding New Features to the Child Class

You have now seen Way 1 — overriding an existing method. Now it is time to give equal attention to Way 2: giving a child class something that the parent never had at all.

✨ What "Adding a New Feature" Means

A child class adds a new feature when it defines a method or a field that does not exist anywhere in the parent. This is not overriding — the parent has nothing to replace. The child is simply extending the parent's abilities with something new.

Contrast Override vs Add New — What Is the Difference?

🔄 Override (Way 1)
  • The method already exists in the parent
  • The child provides its own version of it
  • The parent's version is replaced
  • Same name, same signature — different body
✨ Add New Feature (Way 2)
  • The method does not exist in the parent at all
  • The child creates something brand new
  • Nothing in the parent is replaced or changed
  • Only child objects can call this new method

Code Example A Vehicle That Becomes a Car

Consider a parent class Vehicle that knows how to drive() and stop(). A Car inherits both — and then adds two things Vehicle never had: honk() and openTrunk().

// ── Parent class ────────────────────────────────────── class Vehicle { protected String brand; public Vehicle(String brand) { this.brand = brand; } public void drive() { System.out.println(brand + " is driving."); } public void stop() { System.out.println(brand + " has stopped."); } } // ── Child class ────────────────────────────────────── class Car extends Vehicle { public Car(String brand) { super(brand); } // ✦ NEW — Vehicle has no honk() at all public void honk() { System.out.println(brand + " goes: Beep! Beep!"); } // ✦ NEW — Vehicle has no openTrunk() at all public void openTrunk() { System.out.println(brand + "'s trunk is now open."); } }
Using Car — inherited + new methods together
Car myCar = new Car("Toyota"); myCar.drive(); // ← inherited from Vehicle — works perfectly myCar.stop(); // ← inherited from Vehicle — works perfectly myCar.honk(); // ← NEW — only Car has this myCar.openTrunk(); // ← NEW — only Car has this Toyota is driving. Toyota has stopped. Toyota goes: Beep! Beep! Toyota's trunk is now open.

💡 Notice: drive() and stop() are never redefined in Car — they are simply inherited and used exactly as they are. Only honk() and openTrunk() are new. Car is not replacing anything — it is extending what Vehicle already does.
📌 The Connection to Part 1

You already saw this in Part 1 with EnhancedPrintWriter: Java's PrintWriter has no print2DArray() method. By extending it and adding that one new method, we gave EnhancedPrintWriter an ability PrintWriter never had — while keeping every existing method working perfectly. That is exactly Way 2.

5 Overriding vs Overloading — The Critical Difference

Students frequently confuse overriding and overloading because both involve methods with the same name. The difference is clear once you understand the key question: does the signature change?

Feature Overloading Overriding
Where? Same class Child class — requires inheritance
Method name Same Same
Parameter list Must be different Must be identical
Return type Can be anything Must be same (or covariant)
Resolved when? Compile time Run time
@Override needed? No Recommended — always use it
Keyword used None @Override annotation

Both in One Example Overloading AND Overriding in the Same Subclass

A subclass can both override a parent method AND overload it (add a new version with a different signature). Here is an example that shows both happening at the same time — study it carefully.

class Calculator { public int add(int a, int b) { return a + b; } } class ScientificCalculator extends Calculator { @Override public int add(int a, int b) { // OVERRIDING — same signature as parent System.out.println("Sci: " + a + " + " + b); return a + b; } public double add(double a, double b) { // OVERLOADING — different parameter type return a + b; } public int add(int a, int b, int c) { // OVERLOADING — extra parameter return a + b + c; } } public class Main { public static void main(String[] args) { ScientificCalculator sc = new ScientificCalculator(); System.out.println(sc.add(3, 4)); // calls the OVERRIDDEN version System.out.println(sc.add(1.5, 2.5)); // calls the double OVERLOAD System.out.println(sc.add(1, 2, 3)); // calls the 3-param OVERLOAD } }
Output
Sci: 3 + 4 7 4.0 6

6 Calling the Parent's Version — super.method()

When a child overrides a method, the parent's version is hidden — but not deleted. You can still call it explicitly using super.methodName(). This is especially useful when the child wants to extend the parent's behaviour rather than completely replace it.

💡 When to Use super.method()
  • When the child's version should include everything the parent does, plus more.
  • When you want to avoid duplicating the parent's logic in the child.
  • super.method() can appear anywhere in the child's method body — not just the first line (unlike super() in constructors).
class Shape { private String color; public Shape(String color) { this.color = color; } public String getInfo() { return "Shape[" + color + "]"; } } class Circle extends Shape { private double radius; public Circle(String color, double radius) { super(color); this.radius = radius; } @Override public String getInfo() { // Call parent's getInfo() first, then add Circle's own details return super.getInfo() + " → Circle(radius=" + radius + ")"; } } public class Main { public static void main(String[] args) { Circle c = new Circle("Red", 5.0); System.out.println(c.getInfo()); } }
Output
Shape[Red] → Circle(radius=5.0)
📚 The Pattern

return super.getInfo() + " → " + myExtra is a very common pattern in Java. The child reuses everything the parent computes, then appends its own additional information. No code is duplicated.

7 The protected Modifier

📌
Remember This Promise?

Back in the Access Modifiers topic, we listed four access levels — private, (default), protected, and public — but we deliberately skipped protected and said: "We will explain this one when you reach Inheritance, because you cannot fully understand it without knowing what a subclass is."

You now know what subclasses are. The wait is over.

You know private (visible only inside the same class) and public (visible everywhere). There is a modifier between them — protected — and it exists specifically because of inheritance.

💡 Why We Need protected

The problem: a subclass inherits a parent's fields, but if those fields are private, the subclass cannot access them directly — only through getters/setters.

The solution: mark the field protected so that subclasses can access it directly, while unrelated classes are still blocked.

Visibility Table All Four Access Levels — Compared

Modifier Same Class Same Package Subclass (other package) Everywhere
private ✓ ✗ ✗ ✗
(default / package) ✓ ✓ ✗ ✗
protected ✓ ✓ ✓ ✗
public ✓ ✓ ✓ ✓

Visibility increases from top to bottom: private → (default) → protected → public

💡 Why "other package" — not "any package"?
A student raised a sharp question: the original column said "any package" — but that is misleading. The (default) modifier already allows access to subclasses within the same package. The real superpower of protected is that it additionally lets subclasses in other packages access the member — something (default) cannot do. That is the precise distinction the label "Subclass (other package)" captures.

Example protected in Practice

class Vehicle { private String brand; // private: ONLY Vehicle can touch this protected int speed; // protected: Vehicle AND its subclasses public String color; // public: anyone public Vehicle(String brand, int speed, String color) { this.brand = brand; this.speed = speed; this.color = color; } public String getBrand() { return brand; } } class Car extends Vehicle { private int doors; public Car(String brand, int speed, String color, int doors) { super(brand, speed, color); this.doors = doors; } public void accelerate(int amount) { speed += amount; // ✓ allowed — speed is protected // brand += "X"; // ✗ ERROR — brand is private to Vehicle } @Override public String toString() { return getBrand() + " | Speed: " + speed + " | Color: " + color + " | Doors: " + doors; } } public class Main { public static void main(String[] args) { Car car = new Car("Toyota", 60, "White", 4); car.accelerate(40); System.out.println(car); } }
Output
Toyota | Speed: 100 | Color: White | Doors: 4

Override Accessibility Can a Subclass Change the Access Level of an Overridden Method?

When overriding a method, the subclass has a rule about the access modifier it uses: it cannot reduce the visibility, but it can increase it. Think of it this way — you can open a door wider, but you cannot make it narrower.

✓ Allowed — Increasing Visibility
class Parent { protected void show() { } } class Child extends Parent { @Override public void show() { } // protected → public ✓ }
✗ Not Allowed — Reducing Visibility
class Parent { public void show() { } } class Child extends Parent { @Override protected void show() { } // public → protected ✗ }
⚠ Why Can't Visibility Be Reduced?

Imagine code elsewhere that uses the parent type and calls a public method. If a subclass could make that method private, that existing code would break. Java protects this by enforcing that overridden methods are at least as accessible as the original.

8 The final Keyword

The keyword final means "this cannot be changed". It can be applied to three different things in Java, and in each case the meaning is consistent — something is locked and cannot be modified or extended.

🔒
final Variable
The variable's value cannot be changed after it is assigned. It becomes a constant.
🚫
final Method
The method cannot be overridden in any subclass. Its behaviour is fixed forever.
🏦
final Class
The class cannot be extended. No subclass can ever inherit from it.

final Variable Constants That Cannot Change

class MathConstants { final double PI = 3.14159; // instance constant static final double E = 2.71828; // class-level constant (shared by all) public void demo() { System.out.println("PI = " + PI); // PI = 3; // ← COMPILE ERROR: cannot assign a value to a final variable } }
💡 By convention, final constants are named in ALL_CAPS_WITH_UNDERSCORES. For example: MAX_SPEED, DEFAULT_NAME.

final Method Behaviour That Cannot Be Overridden

class BankAccount { private double balance; public BankAccount(double initial) { this.balance = initial; } // This method is sealed — no subclass can change how deduction works public final void deduct(double amount) { if (amount > balance) throw new IllegalArgumentException("Insufficient funds"); balance -= amount; System.out.println("Deducted: " + amount + ". Balance: " + balance); } } class SavingsAccount extends BankAccount { public SavingsAccount(double initial) { super(initial); } // @Override // public void deduct(double amount) { } // ← COMPILE ERROR: cannot override final }
💡 Why Seal a Method?

Security-critical logic (like financial calculations) should not be alterable by subclasses. Marking a method final guarantees the business rule is always enforced, no matter who extends the class.

final Class A Class That Cannot Be Extended

// Java's String class is declared final in the standard library: // public final class String { ... } // That means you CANNOT do this: // class MyString extends String { } // ← COMPILE ERROR // You can declare your own final classes too: public final class ImmutablePoint { private final double x, y; public ImmutablePoint(double x, double y) { this.x = x; this.y = y; } public double getX() { return x; } public double getY() { return y; } } // class MovingPoint extends ImmutablePoint { } // ← COMPILE ERROR: class is final

9 The Object Class — Java's Ultimate Parent

🙋
A Student Raises Their Hand…

"Professor, I wrote a class with no fields and no methods at all — just an empty class body. But when I created an object, I noticed it had methods like .toString() and .equals(). Where do they come from?!"

📚 The Answer:

That is an excellent observation! Every class in Java automatically inherits from a special class called Object — whether you write it or not. Object is the ultimate parent of every class in the entire Java ecosystem. Those methods you found come directly from it.

// These two are exactly the same to Java: class Student { ... } class Student extends Object { ... } // Java writes this for you invisibly

Key Methods Useful Methods Inherited from Object

Method What It Does Default Behaviour
toString() Returns a text representation of the object ClassName@hexAddress (e.g. Student@1b6d3586)
equals(Object o) Checks equality between two objects Compares memory addresses (same as ==)
hashCode() Returns an integer hash for the object Based on memory address
getClass() Returns the runtime class of the object Cannot override (it is final)

Override toString & equals Making Your Class Meaningful

The default toString() is useless for debugging. Override it to produce meaningful output. Similarly, override equals() when equality should be based on field values, not memory addresses.

class Student { // implicitly extends Object private String name; private int id; public Student(String name, int id) { this.name = name; this.id = id; } @Override public String toString() { return "Student[" + id + "] " + name; } @Override public boolean equals(Object obj) { if (this == obj) return true; // same object in memory if (!(obj instanceof Student)) return false; Student other = (Student) obj; return this.id == other.id; // students are equal if same ID } } public class Main { public static void main(String[] args) { Student s1 = new Student("Ali", 1001); Student s2 = new Student("Ali", 1001); // same ID — different object Student s3 = new Student("Omar", 1002); System.out.println(s1); // calls toString() System.out.println(s1.equals(s2)); // same ID → true System.out.println(s1.equals(s3)); // different ID → false System.out.println(s1.getClass().getName()); // runtime class name } }
Output
Student[1001] Ali true false Student
📚 The Hierarchy Reaches All the Way Up

Object is the root of every hierarchy in Java. This means every object you ever create — String, Scanner, your own classes — is ultimately an instance of Object. This is what makes Java's type system coherent.

10 Multiple Inheritance — and Why Java Says No

In some programming languages, a class can extend more than one parent at the same time. This is called multiple inheritance. Java does not support this — and for a very good reason.

The Problem The Diamond Problem

Imagine three classes: A, B (extends A), and C (extends A). Both B and C override a method from A differently. Now a fourth class D tries to extend both B and C. Which version of the method does D inherit? Java cannot answer this — so it prohibits the question entirely.

A
+ hello() : void
B
+ hello() : void
C
+ hello() : void
D
extends B, C   ✖ COMPILE ERROR
⚠ Java's Decision

Java prohibits extends from listing more than one class. class D extends B, C { } is a compile error. This eliminates ambiguity entirely.

Single Inheritance Hierarchy When a Strict Tree IS the Right Answer

When a strong IS-A relationship exists through a chain of classes, single inheritance is clean and unambiguous. Here is a realistic example:

Person
+ getName() : String
+ getAge() : int
Student
+ getGPA() : double
GradStudent
+ getThesis() : String
Employee
+ getSalary() : double
Professor
+ getDept() : String

Every arrow here is a clear IS-A relationship: a Student IS-A Person, a Professor IS-A an Employee IS-A a Person. This tree is unambiguous — each class has exactly one parent.

When Hierarchy Is Not Enough Strong Relationships vs Weak Relationships

The hierarchy above works perfectly because every link in the chain is a strong IS-A relationship. A GraduateStudent IS-A Student. A Student IS-A Person. No matter how many levels deep the chain goes, the statement "the bottom class IS-A Person" remains true and meaningful. When this condition holds, a clean hierarchy is the right model.

✓ Strong IS-A Chain — Hierarchy Works
Person
→
Student
→
GraduateStudent
→
MasterStudent

Ask at any level: "Is MasterStudent a Person?" — Yes. "Is GraduateStudent a Person?" — Yes. The IS-A test passes all the way up. No multiple inheritance is needed.

But sometimes you want to model capabilities that are shared across objects which do not belong to the same family. Imagine a Duck and a Seaplane — both can fly and both can swim, but they are completely unrelated classes. There is no single meaningful parent that covers both. The IS-A test fails: "Is a Seaplane a Duck?" — No. "Is a Duck a Seaplane?" — No. There is only a weak, functional relationship: they share certain capabilities, not a common identity.

⌛ Weak / Functional Relationship — Hierarchy Breaks Down
Duck
and
Seaplane
both fly and swim — but share no meaningful parent class

Forcing them into one hierarchy would be artificial and misleading. A different Java tool is needed for this kind of relationship.
